FAGS Browser Runtime For PAGS Agents

Runtime model for agents that need to drive a real browser: PAGS owns the control-plane brain, FAGS owns the browser runtime, and the cheapest best-practice target is outbound polling from FAGS to PAGS. Cloudflare Tunnel is the current shipped bootstrap path; managed FAGS runtimes are the later paid reliability path.

Status: implemented tunnel mode Target: outbound polling Runtime: FAGS PAS precedent: our loop Reference agent: job applications

Decision Status

Current code: PAGS stores an instance-scoped FAGS runtime endpoint and proxies task/status/event calls to the FAGS Playwright runtime exposed through Cloudflare Tunnel.

Cheapest best-practice target: FAGS connects outbound to PAGS, polls for approved tasks, executes them locally, and posts events/results back. Managed FAGS runtimes remain the paid reliability path.

System Diagram

PAGS Control Plane

Marketplace, MCP, subscriptions, runtime assignment, task summaries, auth, billing, audit events.

->

FAGS Browser Runtime

Playwright, browser profile, local files, screenshots, uploads, approvals, events, and task results.

->

Real Browser

Logged-in job boards, DOM, screenshots, file uploads, downloads, manual handoff.

PAGS starts and observes work. FAGS executes browser actions. Current code reaches FAGS through a tunnel; target local mode uses outbound polling so the user's machine does not expose a public endpoint.

Current Tunnel Flow

PAGS API/MCP -> Cloudflare Tunnel endpoint -> FAGS browser runtime -> Playwright browser PAGS API/MCP <- task events/status/results <- FAGS browser runtime

Target Polling Flow

PAGS task queue <- poll for work <- FAGS browser runtime -> Playwright browser PAGS task queue -> task events/results -> FAGS browser runtime

Runtime Modes

Hosted Worker

Good for text-only/server-only agents. No Playwright. No real browser session.

Managed FAGS Runtime

Later paid path. Managed FAGS VM/container with Playwright and persistent browser state.

Local FAGS Runtime

Cheapest default target. Node plus Playwright on the user's machine, connected outbound to PAGS. Tunnel mode is shipped today as fallback/bootstrap.

Brain Placement

OptionUpsideProblemRecommendation
Brain in PAGS Worker Centralized orchestration. Browser screenshots, DOM, cookies, and forms must travel through PAGS; high-latency observe/act loop. Use for text-only agents, not browser agents.
Brain in PAGS, runtime in FAGS Cheapest, user-owned browser/files, platform-owned control plane. User machine must be online; polling endpoints still need to be built. Cheapest best-practice recommendation.
Brain in managed FAGS runtime Always-on, managed, supportable, billable, same browser protocol. Costs more and needs stronger isolation/retention policy. Paid reliability path.

Start task with job URL
Receive events and status

FAGS Browser Runtime

Observe page and detect job board
Plan and fill application
Pause for login/captcha/missing answers
Submit only after approval

Protocol Sketch

Agents that need browser automation should declare it in their manifest.

{ "runtime": { "kind": "fags-browser-runtime", "brain": "pags", "browser": { "engine": "playwright", "persistentProfile": true, "manualHandoff": true, "fileUploads": true }, "deployment": { "localPolling": true, "localTunnel": true, "managedVm": true, "hostedWorker": false } } }

Current tunnel runtime endpoints should be signed/tokenized and instance-scoped, not open public tunnel URLs. Target polling mode should use PAGS task queues instead of public local FAGS endpoints.

GET /health GET /capabilities POST /tasks GET /tasks/:id POST /tasks/:id/approve POST /tasks/:id/cancel GET /events
