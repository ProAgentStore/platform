ProAgentStore Browser Runtime

Runtime model for agents that need to drive a real browser: a durable Cloudflare Workflow on PAGS is the brain (using the caller's BYOK Claude), and the local ProAgentStore browser runtime is the hands — real Chrome via Playwright, driven by an ARIA snapshot → act loop. The runtime connects outbound over a WebSocket relay by default (pags up); Cloudflare Tunnel is a legacy/optional path. The flagship user is the LLM-driven job-apply agent.

Brain: JobApplyWorkflow (BYOK Claude) Hands: local browser runtime Default: WebSocket relay Legacy: Cloudflare Tunnel Flagship: job applications

Status

Shipped: the job-apply brain is a durable Cloudflare Workflow (JobApplyWorkflow) that runs on PAGS using the caller's BYOK Claude. Being durable and resumable, it escapes the 30s Durable Object request limit. The brain never touches the browser directly — it drives the local ProAgentStore browser runtime (real Chrome via Playwright) through an ARIA snapshot → act loop: snapshot the accessible tree, let Claude pick one action, act by role + accessible name (no CSS selectors), repeat.

Connectivity: pags up connects the runtime outbound over a WebSocket relay by default — no tunnel binary, nothing public exposed, and one runner multiplexes every one of your agents. Cloudflare Tunnel (--tunnel quick|named) remains as a legacy/optional path.

Connectivity Modes

WebSocket Relay (default)

pags up. No tunnel binary and nothing public exposed — the runtime dials out to RelayDO with a short-lived, instance-scoped relay token. One runner serves all your agents.

Cloudflare Tunnel (legacy)

pags up --tunnel quick|named. Quick tunnels are ephemeral and rate-limited; named tunnels give a stable hostname. Kept for compatibility/debug.

Hosted Worker

For text-only/server-only agents that need no browser. No Playwright, no local runtime, no pags up.

Brain — JobApplyWorkflow PAGS, as a durable Cloudflare Workflow ([[workflows]] JOB_APPLY) on the caller's BYOK Claude. Reads the ARIA snapshot, picks one action per step, decides handoffs, applies per-ATS tips, reads confirmation emails. Durable + resumable, so it survives the 30s DO request limit and long applications. The apply loop (lib/apply-loop.ts) is pure and tested.
Hands — browser runtime The user's own machine, launched by pags up. Runs real-profile Chrome via Playwright; exposes /browser/snapshot + /browser/act and the takeover endpoints. The browser session, cookies, and files stay on the user's machine; only the accessible tree and actions cross the wire.
Guard — dry run Inside the Workflow. When dryRun:true, a runtime guard blocks the final submit click; the brain cannot override it. Lets you fill an entire application safely without submitting for real.

Apply Loop & Handoffs

Trigger with POST /v1/instances/:id/apply { url, resumePath, dryRun? }. The brain loops snapshot → decide → act until the application is submitted (or blocked). It reads the structured Profile + Special Instructions + the per-ATS learned-tips cache, and is hard-locked to use a known value or ask — never to invent one.

Raise a handoff when it can't proceed

captcha — solve it in the live takeover; auto-resumes when the token appears
stuck — after repeated fails on a widget, you do that one step and Resume
needs_input — a required value the agent lacks; console shows an input box, value is saved to the Profile, run resumes
Confirmation emails surface as "Open in Gmail" links

Runtime Protocol

Agents that need browser automation declare a browser runtime in their manifest. The runtime identifies itself as ProAgentStore on its /health (service: "proagentstore-browser-runtime") and /capabilities (runtime: "pags-browser-runtime", runtimePlane: "pags").

{ "runtime": { "kind": "pags-browser-runtime", "brain": "pags", "browser": { "engine": "playwright", "persistentProfile": true, "manualHandoff": true, "fileUploads": true }, "connectivity": { "relay": true, "tunnel": "legacy", "hostedWorker": false } } }

The runtime exposes an act-by-accessibility surface (no CSS selectors) plus health/capability probes and the live takeover endpoints. The relay connection is authenticated with a short-lived, instance-scoped relay token — never the long-lived account JWT.

GET /health # service: proagentstore-browser-runtime GET /capabilities # runtime: pags-browser-runtime, runtimePlane: pags POST /browser/snapshot # ARIA accessibility tree POST /browser/act # act by role + accessible name (type/select/check/click) GET /takeover* # live remote-control takeover (captcha / stuck / needs_input)

Roadmap

  • Shipped: JobApplyWorkflow brain on BYOK Claude, the ARIA snapshot/act runtime, WebSocket relay as the default connectivity, the three human handoffs, the per-ATS learned-tips cache, and "Open in Gmail" confirmation links.
  • Shipped: pags up multiplexes all of a user's agents through one runner and one relay; Cloudflare Tunnel kept as a legacy/optional path.
  • Next: remove the remaining tunnel code paths (CLI + API) now that relay is the default.
  • Next: harden the credentials/Profile vault (zero-knowledge / KMS / audit log) and add per-instance consent over private Profile fields.
  • Next: application rate-limit + dedup, and auto-search/discovery so the agent can find postings rather than only applying to a URL you give it.
