July 6, 2026

LLM-driven job-application agent

Feature

Give the job-application agent a job URL and an LLM drives a real browser to fill and submit the application — no selectors, it acts on what it sees.

  • Submit for real: the brain fills the ATS form and clicks Submit (with a dry-run guard for testing)
  • Retry + attempts: one card per job on the work board, each attempt tracked and re-runnable
  • Human takeover: captcha / needs-input handoffs pause the run, you resolve it, the agent resumes
  • Per-ATS learned tips: every run saves what worked and failed, keyed by ATS host, fed back next time
  • Confirmation email: the employer's confirmation is recorded as an "Open in Gmail" link
  • Profile + Job Preferences: structured, reusable candidate details answer application questions — the agent never invents values
July 6, 2026

Coder + Repo Chat

New Agents

Two new first-party agents that work with your code.

  • Coder: an AI coding agent that runs a coding CLI — Claude Code, Codex, or Grok — on your own machine via pags up, driving any GitHub repo. Live terminal, Co-pilot summaries, autonomous Loop, and a cross-repo Overseer.
  • Repo Chat: point it at any GitHub repo — it indexes the whole codebase into a vector store and you chat with it. Read-only by construction.
July 6, 2026

Work board, Markdown docs, voice, observability

Feature
  • Agent-configurable work board: one kanban board per instance, columns declared by the agent, one card per job — move cards, retry, and drill into attempts
  • First-class Markdown documents: create, read, edit, and render docs in the Knowledge base; the agent can read and update them from chat
  • Voice hands-off mode: Whisper AI transcription with adjustable Mic Sensitivity and pause length — push-to-talk, auto-speak, or full conversation
  • Error observability: browser and server failures land in a durable log, readable via MCP list_errors
July 6, 2026

pags up + WebSocket relay CLI

Feature
  • One runner for all your agents: npm i -g @proagentstore/cli then pags up — a single runner serves every active instance
  • WebSocket relay: the runner connects outbound over a relay, so there's no tunnel binary and no config; short-lived, instance-scoped relay tokens per connection
  • Multi-CLI engines: each coding session launches your chosen engine (Claude Code / Codex / Grok)
  • Machine takeover: one machine per instance, with pags up --force to take over; suspended sessions resume on reconnect
